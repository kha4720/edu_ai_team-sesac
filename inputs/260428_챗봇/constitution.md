# Constitution — 질문 코칭 챗봇 교수 설계서

> **문서 성격**: 0차 산출물 (헌법). 모든 후속 산출물의 교육적 판단 기준이자 Orchestrator의 Gate 검증 기준.
> **작성 주체**: Edu Agent
> **버전**: v1.0 (Layer 2 입력용 — 질문 코칭 챗봇)
> **출처**: 김현아 님 기획서 v2 5.2장 추출

---

## A. 설계 근거

### 1. 교육공학적 문제 재정의

| 현재 문제 | 근본 원인 | 학습 결핍 정의 |
| --- | --- | --- |
| 학생들이 AI에 질문을 잘 하지 못한다. | 좋은 질문의 구조를 모른다. | **질문 구성 능력의 부족** |

표면적으로는 "AI 사용법을 모른다"로 보이는 문제이지만, 본질은 **자기가 무엇을 묻고 싶은지를 언어로 구조화하는 능력의 부족**이다. 따라서 본 서비스는 AI 사용법 강의가 아니라, 학생이 자기 질문을 다듬는 반복적 훈련 경험을 제공해야 한다.

### 2. 교수 기법 탐색

| 교수 기법 | 기대 효과 | MVP 적용 난이도 |
| --- | --- | --- |
| **산파법 (Socratic Method)** | 답을 바로 주지 않고 되묻기로 학생이 스스로 깨닫게 함 | 낮음 (프롬프트 중심 구현 가능) |
| **스캐폴딩 (Scaffolding)** | 한 번에 정답을 주지 않고 단계별 발판 제공 | 낮음 (프롬프트 중심 구현 가능) |
| **QFT (Question Formulation Technique)** | 거친 질문을 대상·주제·목적·요청 형태로 분해해 명확화 | 중간 (간단한 상태 관리 필요) |
| **메타인지 유도** | 자신이 무엇을 알고 모르는지 스스로 점검 | 중간 (간단한 상태 관리 필요) |
| **PBL (문제 기반 학습)** | 실제 문제 해결 프로젝트 중심 학습 | 높음 (장기 운영 필요) |

### 3. MVP 교수 기법 선정

**채택 기법: 산파법 + 스캐폴딩**

채택 이유: 대화형 챗봇 구현에 가장 적합하며, 프롬프트 중심으로 즉시 구현 가능하다. 산파법은 되묻기 메커니즘의 근거가 되고, 스캐폴딩은 한 번에 하나씩 보완하는 단계적 구조의 근거가 된다.

**제외 기법: PBL**

제외 이유: 프로젝트 단위 운영이 필요해 MVP 범위를 초과한다.

## B. 실행 기준 (하네스 헌법)

### 4. 서비스 전체 설계 원칙

본 서비스는 다음 네 가지 원칙으로 작동한다.

1. **답을 바로 제공하지 않는다.** 정답을 주는 것이 아니라, 학생이 자기 질문을 다듬는 경험을 쌓게 만든다.
2. **부족한 요소를 한 번에 하나씩 순서대로 보완하게 한다.** 여러 부족점을 동시에 지적하지 않고, 가장 우선되는 한 가지만 짚는다. (피드백 방식: **순차적 보완**)
3. **기준 충족 시 즉시 다음 단계로 이동한다.** 불필요하게 학습을 길게 끌지 않는다.
4. **평가보다 자기 수정 경험을 우선한다.** 학생이 위축되지 않도록 평가적 표현을 피하고 성장적 표현을 쓴다.

### 5. 학습 목표 (Learning Objective)

> 학생이 AI에게 질문할 때, **주제** / **상황** / **원하는 답의 형태**를 포함해 스스로 질문을 작성할 수 있다.

- **주체**: 학생
- **조건**: AI에게 질문하는 상황에서
- **관찰 가능한 행동**: 주제 / 상황 / 원하는 답의 형태 세 요소를 포함한 질문을 스스로 작성

### 6. 평가 루브릭 (Rubric)

| 평가 기준 | 우수 | 양호 | 미흡 |
| --- | --- | --- | --- |
| **구체성** (주제/대상) | 주제/대상이 명확하고 범위가 좁혀져 있음<br>예) "'내 마음은 호수요'가 왜 비유 표현인지" | 주제는 있으나 범위가 모호함<br>예) "비유가 뭔지" | 주제/대상이 전혀 없음<br>예) "국어 모르겠어요" |
| **맥락성** (상황) | 상황이 구체적으로 명시됨<br>예) "국어 숙제인데" | 상황이 암시되어 있으나 불분명함<br>예) "수업 시간에 나온 건데" | 상황이 전혀 없음 |
| **목적성** (원하는 답의 형태) | 원하는 답의 형태가 명확함<br>예) "이유를 설명해줘" | 원하는 것이 있으나 형태가 모호함<br>예) "도와줘" | 원하는 답의 형태가 전혀 없음 |

**최소 성취 기준**: 3가지 기준 모두 양호 이상

**설정 이유**: 본 서비스는 "학생이 스스로 깨닫게 하는 것을 우선(④)"하므로 전 기준 양호 이상을 최소 조건으로 설정한다. 한 영역이 미흡한 상태로 학습 목표를 달성한 것으로 처리하면 메타인지 향상 효과가 약해진다.

### 7. 루브릭 기반 서비스 플로우 원칙

**평가 방식**: 순차 평가 방식

**선택 근거**: ④의 "부족한 요소를 한 번에 하나씩 순서대로 보완하게 한다" 원칙에 따라 순차 평가 방식을 선택한다.

**판단 순서**: 구체성 → 맥락성 → 목적성

**판단 순서 근거**: 구체성(무엇에 대한 질문인가)이 확보되지 않으면 맥락성(왜 필요한가)과 목적성(어떤 도움이 필요한가)을 판단하는 것 자체가 불가능하기 때문에 가장 먼저 평가한다. 맥락성은 목적성보다 넓은 조건이므로 목적성보다 먼저 평가한다.

| 조건 | 시스템 판단 | 시스템 출력 | 사용자 다음 경험 |
| --- | --- | --- | --- |
| 구체성 양호 이상 | 맥락성 판단으로 이동 | 별도 출력 없음 | 대기 |
| 구체성 미흡 | 구체성 보완 유도로 이동 | 구체성 보완을 유도하는 되묻기 출력 / 재입력창 활성화 | 질문 보완 후 재제출 |
| 맥락성 양호 이상 | 목적성 판단으로 이동 | 별도 출력 없음 | 대기 |
| 맥락성 미흡 | 맥락성 보완 유도로 이동 | 맥락성 보완을 유도하는 되묻기 출력 / 재입력창 활성화 | 질문 보완 후 재제출 |
| 목적성 양호 이상 | 완료 플로우로 이동 | 개선된 질문 + 칭찬 피드백 출력 | 결과 화면 확인 후 종료 또는 재시도 |
| 목적성 미흡 | 목적성 보완 유도로 이동 | 목적성 보완을 유도하는 되묻기 출력 / 재입력창 활성화 | 질문 보완 후 재제출 |

## 우선순위 원칙

1. 본 헌법은 0차 산출물로서 모든 후속 문서보다 우선한다.
2. 하위 산출물이 본 헌법과 충돌할 경우 헌법 기준을 우선 적용한다.
3. 서비스 변경 시에도 본 헌법(특히 ④~⑦)과의 정합성을 우선 검토한다.
