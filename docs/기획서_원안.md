# **2\. 기획 문서 하네스 구축**

### **2\. AI 기획팀: 기획 문서 하네스(platform harness) 구축** 본 AI Agent 팀(Platform Harness, Layer 1층, 이하 기획 문서 하네스)은 **교육 서비스 회사의 사내 구성원**(인간)에게 **AI를 기반으로 하는 교육 서비스 아이디어를 입력**받아, **교육 특화 Agent 간의 협업**을 통해 **MVP 설계 문서를 생산하는 시스템**이다.

### **2.1. 핵심 기능**

* 사용자 아이디어를 교육 서비스 문제로 재해석  
* 교육 운영 원칙(헌법) 수립  
* 서비스 기획 문서 생성  
* 구현 명세서 생성  
* 범용 AI 기반 MVP 데모 제작 지원

### **2.2. 산출 목표**

* 사람이 바로 검토 가능한 문서 패키지  
* 범용 AI가 구현 가능한 명세서  
* 단기간 MVP 데모 제작

### **2.3. 서비스 플로우 예시**

* 본 하네스는 사용자 Input과 Constitution을 전역 기준(Global Context)으로 유지하며 작동한다.  
* 각 Agent는 직전 산출물뿐 아니라 Input(서비스 기획 정보, 실행 제약 정보), Constitution(교육 운영 원칙)을 함께 참조하여 산출물 간 정합성을 유지한다.


| 단계 | 구분 | 주요 내용 | 산출물 / 예시 |
| ----- | ----- | ----- | ----- |
| 1 | 사용자 Input | AI 활용을 전제로 한 교육 서비스 아이디어와 실행 제약 정보를 입력 | 서비스 기획 정보: 타깃 유저 / 문제 상황 / 서비스 목표 / 솔루션 아이디어 실행 제약 정보: 마감 기한 / 팀원 구성 / 기술 수준 / 기존 자산 여부 |
| 2 | \[Process 1\] 헌법 생성 | 교육 도메인 지식에 특화된 sub Agent가 교육공학 지식을 바탕으로 교육 운영 원칙(헌법) 수립 | 서비스의 기준점이 되는 헌법 문서 (constitution.md) |
| 3 | \[Process 2\] 기획 문서 5종 (사람용 문서) | 서비스 방향 합의 및 팀 내 협업을 위한 기획 문서 5종 생성 | Service Brief.md / MVP Scope.md / User Flow.md / Build Plan.md / QA Plan.md |
| 4 | \[Process 3\] 구현 명세서 4종(AI용 문서) | 범용 생성형 AI가 구현 가능한 수준의 구현 명세서 4종 | data\_schema.json / state\_machine.md / prompt\_spec.md / interface\_spec.md  |

※ 사용자 Input값에 대한 구체적 설명은 4장, 산출물에 대한 구체적인 설명은 5장 참고

### **2.4. 기획 문서 하네스 MVP 검증 및 활용 예시** 

기획 문서 하네스의 실효성을 검증하기 위해 **중학생 질문력 향상 챗봇**을 사례 과제로 선정한다.  
하네스를 통해 **서비스 기획 문서와 구현 명세서를 생산**하고, 이를 **범용 생성형 AI에 입력하여 MVP Demo를 구현**한다.

구현된 결과물은 실제 운영 중인 질문력 에이전트와 비교하여 아래 항목을 중심으로 검증한다.

* 구성 요소의 적절성  
* 핵심 기능 흐름의 완성도  
* 문서 기반 구현 가능성  
* MVP 수준의 실행 가능성

또한 본 하네스는 동일 방식으로 다양한 형태의 서비스 MVP 제작에 활용될 수 있다.

활용 예시:

* 웹앱 프로토타입  
* 챗봇 MVP  
* 클릭형 데모 서비스  
* 내부 PoC 검증용 서비스

### **2.5. 의도적 제외 범위**

- 본 하네스는 문서 생산 및 MVP 설계 자동화 시스템이므로 아래 항목은 현재 범위에서 제외한다.  
* 시장 검증  
* 실사용 UX 검증  
* 비즈니스 모델 검증  
- 위 항목들은 본 하네스를 통해 제작된 MVP를 활용해 후속 단계에서 검증할 내용이라 판단.

### **3.1 헌법 — Edu Agent가 만드는 교수 설계서**

* Edu Agent가 웹 서치 기반으로 작성하는 **최상위 교육 설계 문서**이다.  
* 서비스 기획 문서 5종 및 구현 명세서 4종의 **타당성 판단 기준**으로 활용된다.


| 항목 | 내용 |
| :---- | :---- |
| **④ 서비스 전체 설계 원칙** | MVP 완성을 위해 선택된 교수 기법의 핵심 원리를 서비스 동작 철학으로 번역한 것 |
| **⑤ 학습 목표 (Learning Objective)** | 타깃 유저가 이 서비스를 통해 달성해야 할 상태 |
| **⑥ 평가 루브릭 (Rubric)** | 학습 목표 달성 수준을 실시간으로 판단하는 기준 |
| **⑦ 루브릭 기반 서비스 플로우** | 루브릭 판단 결과에 따라 서비스가 어떻게 동작하는가 |

※ 교수 설계서 전체 항목 및 상세 내용은 **5.2장**에서 별도 설명한다.

### **3.2. 작업 구조 — Agent 간단 소개**

| **Edu Agent** | 교육공학 기반으로 서비스를 설계하고 교육적 효과와 사용자 경험을 동시에 판단하는 교육 서비스 기획 전문가 | 교수 설계서(헌법) 수립 기획 문서 5종의 헌법 준수 여부 검토 구현 명세서 4종의 헌법 준수 여부 검토 |

| **Tech Agent** | 기술적 구현 가능성과 품질을 동시에 책임지는 테크 리더 | 기획 문서 중 Build Plan 작성 기획 문서 5종 및 구현 명세서 4종의 기술적 타당성 검토 구현 명세서 중 구현 명세서 단계에서 pytest 코드 생성 |

### **3.3. 검증 구조**

* 본 하네스의 게이트는 총 4개로 Orchestrator가 총괄함.  
* 도메인 판단이 필요할 때는 전문 Agent의 결과를 수합해 Orchestrator가 최종 결정.

| Gate | 위치 | Orchestrator 직접 판단 | 서브 Agent 의견 수합 |
| ----- | ----- | ----- | ----- |
| **Gate 1** | 교수 설계 직후 | 헌법 완전성 / 유효성 / 정합성 / 근거성 | Tech: 실행 가능성 |
| **Gate 2** | 서비스 기획 문서 5종 작성 후 | 5종 문서 간 정합성 | Tech: 기술 실행 가능성 / PM: Build Plan 타당성 / Edu: 헌법 반영 여부 |
| **Gate 3** | 구현 명세서 4종(pytest 제외) 작성 후 | 4종 명세서 정합성 / 완전성 | Tech: 구현 가능성 Edu: 헌법 훼손 여부 |
| **Gate 4** | pytest 코드 작성 후 | 문법 오류 / 커버리지 / 필드 일치 / assert 기준과 루브릭 일치 여부 | 없음  (Orchestrator 단독 판단) |

#### **3.5. 기록 구조(REPOSITORY)**

* 본 하네스에서 생성되는 운영 원칙, 산출물, 검토 결과, 의사결정 로그를 저장하고 재사용하기 위한 구조(notion, github, local 등 다양한 저장 장소 활용 가능)  
* 기록 구조는 이후 동일한 유형의 교육 서비스 기획을 반복 생성할 때 재사용되는 기준 자산으로 기능함.  
* 다음과 같은 기록 구조를 가질 수 있음.  
  * 운영 원칙: AI가 항상 따라야 하는 공통 기준  
  * 산출물 템플릿: 문서 생성 시 재사용하는 기본 양식  
  * 프로젝트별 결과물: 생성된 기획 문서 및 버전 관리  
  * 리뷰 결과 기록: 하네스 內 각 agent들의 검토 의견  
  * 의사결정 로그: 각 agent들이 왜 그렇게 결정했는지 남기는 기록  
  * 실패·성공 사례: 반복하면 안 되는 오류 / 효과적인 패턴  
  * 용어 사전: 문서 간 표현을 통일하기 위한 기준어 목록  
  * 실행 로그 / 에러 로그: MVP 테스트 결과 / 오류 내역  
  * 재사용 자산: 프롬프트 / 체크리스트 / 운영 문서

### **4.2. Workflow State schema(워크플로우 상태 스키마)**

* 하네스는 **Step(산출물 생성 단계)**과 **Gate(검증 단계)**가 반복되는 구조   
  → 현재 진행 상태·검증 결과·수정 책임의 구조적 관리가 필요함.  
* 본 스키마는 하네스의 실행 상태를 저장함.  
* 본 스키마의 최종 목적은 **구현 명세서 4종 \+ pytest 코드 패키지 완성**임

#### **4.2.1. 핵심 필드 정의                                                           (관련 문서: 3.4.2. 워크플로우 단계)**

| 구분 | 필드명 | 설명 |

#### **4.2.5. Workflow State 관리 규칙**

##### **① current\_task 기록 범위**

* current\_task는 메인 산출물 생성·수정 작업만 기록한다.  
* 검토 의견 작성, 메모 기록 등 Gate 결과물은 current\_task로 기록하지 않는다.  
* 예시: mvp\_scope\_write,interface\_spec\_write, pytest\_write

##### **② Gate 결과물 관리 원칙**

* 검토 단계에서 발생하는 결과는 **Gate 판정값 및 메모 필드**로 관리한다.  
* Review Memo, Feedback Memo 등은 별도 Task로 분리하지 않는다.  
* 예시:  gate\_result \= fail → feedback\_memo \= MVP 범위 축소 필요  
            gate\_result \= conditional\_pass → risk\_memo \= 일정 초과 가능성 있음

##### **③ Fail 발생 시 처리 원칙**

* Gate 검증 결과가 fail일 경우 아래 상태로 전환한다.  
  gate\_result \= fail  
  task\_status \= reworking  
  feedback\_target \= 수정 대상 Agent  
  feedback\_memo \= 수정 요청 내용  
* 수정 대상 Agent는 기존 산출물을 보완한 뒤 동일 Gate 재검토를 진행한다.

##### **④ 재검토 한도 및 조건부 통과**

* **동일 Gate는 최대 2회(초기 검토 1회 \+ 재검토 1회)까지 수행**한다.  
* 재검토 후에도 승인되지 않을 경우 **추가 검토 없이 conditional\_pass로 전환**한다.  
* conditional\_pass 전환 시 **risk\_memo를 반드시 기록하고 다음 단계로 진행**한다.

##### **⑤ Task 실행 원칙**

* 모든 Task는 **선행 Task 완료 후 다음 Task 진행**을 원칙으로 한다.  
* 기본 실행 방식은 순차 처리이다.

##### **⑥ 병렬 실행 예외 (Step 4\)**

* Step 4(구현 명세서 생성)의 일부 Task는 상호 의존성이 낮아 병렬 실행 가능하다.  
* 병렬 가능 Task:  
* data\_schema\_write  
* state\_machine\_write  
* prompt\_spec\_write  
    
* interface\_spec\_write는 아래 Task 완료 후 수행한다.  
  * data\_schema\_write  
  * state\_machine\_write

##### **⑦ 상태 기록 최소 원칙**

* Workflow State는 **현재 위치, 현재 작업, 검증 결과, 예외 이력** 중심으로 기록한다.  
* 세부 사고 과정이나 중간 로그는 상태값으로 관리하지 않는다. 

### **4.3. Artifact Schema(산출물 스키마)**

#### **4.3.1. 산출물 스키마의 필요성**  

* 하네스가 생성해야 하는 **산출물**의 목록, 작성 주체, 입력값, 의존 관계, 필수 목차, 검증 기준을 구조화해서 관리하기 위한 메타 스키마다  
* 문서를 직접 작성하는 문서가 아니라, 문서를 어떻게 생성·검증·관리할지 정의하는 운영 기준표이다.  
* 하네스의 산출물은 단계별로 서로 의존되어 연결(예: service\_brief → mvp\_scope)되지만, 이 구조를 **자연어로만 관리**하면 **어떤 문서를 먼저 생성해야 하는지 혼란이 생겨** **하네스 운영에 차질**이 생김.   
* Artifact Schema는 위 내용을 구조화하여, 생성 순서 고정 / 산출물 책임 주체 명확화 / 입력값 누락 방지 / 문서 품질 표준화 / Gate 자동 검증 지원 / 워크플로우 자동화 가능 상태를 만듦.

####  **4.3.2. 핵심 필드 정의** 

| 필드 | 설명 | 필요한 이유 |
| ----- | ----- | ----- |
| artifact\_id | 산출물 고유 ID | 코드와 워크플로우 식별용 |
| file\_name | 실제 생성 파일명 | 저장 및 출력 연결 |
| stage | 생성 단계 번호 | 순서 제어 |
| category | 문서 유형 | 기준 문서 / 기획 문서 / 구현 명세 구분 |
| owner\_agent | 작성 책임 Agent | 역할 명확화 |
| definition | 한 줄 정의 | 문서 목적 고정 |
| inputs | 참조 입력값 | (Global/Primary/Secondary) 데이터 참조 값 정의  global: 거의 모든 문서에 공통 반영되는 기준값 primary: 해당 문서 작성의 핵심 근거 secondary: 보조 참고 자료 |
| dependencies | 선행 산출물 | 생성 가능 조건 판단 |
| required\_sections | 필수 목차 | 누락 방지 |
| writing\_rules | 작성 원칙 | 품질 표준화 |
| quality\_criteria | 검증 기준 | Gate 판단 자동화 |
| output\_format | 파일 형식 | md/json 구분 |

####  **4.3.3. Orchestrator 활용 방식**

* **생성 단계 판단:** 지금 생성해야 할 문서는 무엇인가? / 선행 문서가 완료되었는가? / 담당 Agent는 누구인가?  
* **검증 단계 판단:** 필수 목차가 모두 존재하는가? / 입력 기준을 반영했는가? / 다른 문서와 충돌하지 않는가? / 다음 단계 진행이 가능한가?  
* **수정 단계 판단:** 어떤 Agent에게 피드백을 보내야 하는가? / 어느 필드 기준으로 FAIL 처리할 것인가?

#### **4.3.4. artifact schema 운영 원칙** 

**① 문서는 자유 산출물이 아니라 관리 대상이다:** 모든 문서는 명확한 생성 조건과 검증 기준을 가져야 함.  
**② 선행 문서 없는 후속 문서는 생성하지 않는다:** 의존성(Dependencies)이 충족되어야만 다음 단계 워크플로우로 이동함.  
③ **Agent 역할은 문서 단위로 고정한다:** 담당자를 명확히 하여 책임 혼선을 방지함.  
       **④ Gate 기준은 구조화한다:** 품질 판단을 주관이 아닌 객관적 데이터 구조에 기반하여 수행함.

#### **4.3.5. artifact schema 구조 및 분류 체계**

| 구분 | 권장 카테고리 체계 (Category) | 예시 스키마: Service Brief (Field-Level)   |
| :---- | :---- | :---- |
| **최상위 기준** | **Constitution** (헌법/원칙) | artifact\_id: "service\_brief" |
| **기획 문서군 (Planning)** | service\_brief (요약) mvp\_scope (범위) user\_flow (흐름) build\_plan (구축 계획) qa\_plan (검증 계획) | owner\_agent: "pm\_agent" definition: "서비스 핵심 개념 요약 문서" stage: 3 (단계 번호) |
| **구현 명세군 (Implementation)** | data\_schema (데이터 구조) state\_machine (상태 정의) prompt\_spec (프롬프트 상세) interface\_spec (인터페이스) | inputs: global/primary/secondary 구조화 dependencies: \["constitution"\] |

### **5.2. 교수 설계서(constitution)**

#### **5.2.1. 목적 정의**

* Constitution은 하네스의 최상위 교육 설계 기준서로 다음의 역할을 수행한다.  
  * 서비스가 왜 그렇게 작동해야 하는지 설명하는 기준서  
  * 학습 목표와 평가 기준을 정하는 설계서  
  * 하네스의 후속 산출물(기획 문서 5종와 구현 명세서 4종)의 판단 기준  
  * Orchestrator의 Gate 검증 기준

#### **5.2.2. 교수 설계서 생성 절차**

* Edu Agent는 아래 절차를 통해 교수 설계서(Constitution)를 생성한다.  
* ①\~③: 최종 헌법을 만들기 위한 사전 설계 단계  
* ④\~⑦: 기획 문서에서 직접 참조하는 핵심 기준, 즉 하네스의 헌법

| 구분 | 단계 | 항목 | 목적 |
| ----- | :---: | ----- | ----- |
| 설계 근거 | ① | 교육공학적 문제 재정의 | 사용자 문제를 학습 결핍 관점에서 재해석 |
|  | ② | 교수 기법 탐색 | 문제 해결에 적합한 교수 기법 후보 조사 |
|  | ③ | MVP 교수 기법 선정 | 실행 제약을 고려해 핵심 교수 기법 선택 |
| 실행 기준(하네스 헌법) | ④ | 서비스 전체 설계 원칙 | 서비스가 작동하는 교육 철학 정의 |
|  | ⑤ | 학습 목표 (Learning Objective) | 사용자가 도달해야 할 행동 변화 정의 |
|  | ⑥ | 평가 루브릭 (Rubric) | 학습 목표 달성 수준 판단 기준 정의 |

#### **5.2.6. \[Gate 1\] 헌법 검증 기준**

* **Gate 1:** 교수 설계서(Constitution)가 후속 문서의 기준으로 사용 가능한지 검증하는 단계

| ----- | ----- | ----- |

|  | 내부 연계성 | ⑤\~⑦이 논리적으로 연결되어 있는가? |
|  | 근거 타당성 | ①\~③ 판단이 ④에 적절히 반영되었는가? |

* **Gate 판정 결과**  
  * **PASS:** 3단계 서비스 기획 문서 작성으로 이동  
  * **FAIL:** Orchestrator가 Feedback Memo 작성 후 Edu Agent에게 수정 요청  
  * **CONDITIONAL PASS:** 재검토 후 동일 문제 발생 시 Risk Memo 남기고, 다음 단계로 이동

#### **5.3.6. \[Gate 2\] 서비스 기획 문서 5종 검증 기준**

* 서비스 기획 문서 5종(Service Brief, MVP Scope, User Flow, Build Plan, QA Plan)이

* Orchestrator가 최종 판정하며, 필요 시 Tech Agent, Edu Agent의 검토 의견을 수합함.

**GATE 판정 결과**

* PASS → 4단계 구현 명세서 작성으로 이동  
* FAIL → Ochestrator가 Feedback Memo 작성 후 해당 Agent 수정 요청  
* CONDITIONAL PASS → 동일 문제 반복 시 Risk Memo 기록 후 다음 단계 진행 검토

### **5.4. 구현 명세서 4종**

#### **5.4.1. Data Schema**

| 항목 | 내용 |

| **작성 담당 Agent** | PM Agent |
| **주요 Input** | \[Global\] Constitution / \[Primary\] MVP Scope, User Flow / \[Secondary\] Build Plan |
| **작성 원칙** | **1\. 작성 기준 (핵심)** 모든 필드는 반드시 User Flow의 특정 단계에서 필요해야 한다. MVP Scope의 "필수 기능"을 수행하는 데 필요한 데이터만 포함한다. MVP 구현에 필요한 최소 필드만 정의한다. Constitution ⑦ 루브릭 기반 서비스 플로우에 조건별 시스템 판단이 정의된 경우, 현재 사용자 상태와 다음 서비스 동작을 함께 표현하기 위해 mode 필드를 반드시 정의한다. **2\. mode 필드 정의 규칙** mode는 단순 상태값이 아니라 "현재 사용자 상태 \+ 다음 서비스 동작"을 함께 나타낸다. mode 값은 ⑦의 시스템 판단 중 사용자에게 노출되는 상태만 포함한다. 내부 이동(예: 다음 기준으로 이동)은 mode 값으로 정의하지 않는다. mode 값은 화면 표시 방식과 다음 사용자 행동을 결정할 수 있어야 한다. **3\. 포함할 데이터** 사용자 입력값 AI 또는 시스템 출력값 화면에 표시되는 데이터 사용자 행동(버튼, 다음 단계)에 영향을 주는 데이터 사용자 경험과 직접 연결되지 않는 내부 처리 정보는 포함하지 않는다. **4\. 필드 정의 방식** 각 필드는 다음을 포함한다: type: 데이터 타입 required: 필수 여부 description: 필드 의미 및 조건부 동작 명시(예: 특정 mode일 때 null 등) allowed\_values (필요 시) example: 예시 값 source: User Flow 단계 또는 루브릭 기반 서비스 플로우 기준 **5\. 활용 방식** 이 스키마는 API 응답 구조와 화면 데이터 기준으로 사용된다. mode는 UI 렌더링과 다음 행동 흐름을 제어하는 기준으로 사용된다. next\_action은 mode와 1:1 대응되며, 프론트엔드의 구현 오류 방지를 위해 명시적으로 정의한다. 실제 DB 구조나 내부 구현은 개발 단계에서 별도로 설계한다.  |
| **주요 판단 기준** | MVP 기능 구현에 필요한 최소 데이터 포함 여부(필수 기능 수행에 불필요한 필드가 없는가) / API 및 처리 로직과의 연결성(각 필드가 실제 요청/응답 흐름과 연결되는가) / 필드 정의 명확성(type, required, description, example이 모두 작성되었는가) / 구현 가능성(정의된 필드가 Build Plan의 기술 방식으로 구현 가능한가) |
| **산출물 이름** | data\_schema.json |
| **예** | {   "input": {     "user\_question": {       "type": "string",       "required": true,       "description": "사용자가 입력한 원래 질문",       "example": "국어 수행평가 질문을 어떻게 써야 할지 모르겠어요.",       "source": "User Flow \- 질문 입력 단계"     }   },   "output": {     "mode": {       "type": "string",       "required": true,       "allowed\_values": \[         "need\_specificity",         "need\_context",         "need\_purpose",         "completed"       \],       "description": "루브릭 판단 결과에 따라 현재 사용자 상태와 다음 서비스 동작을 나타내는 값. 사용자에게 노출되는 상태만 포함하며 내부 이동 상태는 제외.",       "example": "need\_specificity",       "source": "Constitution ⑦ 루브릭 기반 서비스 플로우"     },     "diagnosis": {       "type": "string",       "required": true,       "description": "사용자 입력에서 부족한 점에 대한 진단",       "example": "질문하려는 주제나 대상이 아직 분명하지 않아요.",       "source": "User Flow \- 질문 진단 단계"     },     "follow\_up\_question": {       "type": "string",       "required": false,       "description": "부족한 요소를 보완하도록 유도하는 되묻기. mode가 completed일 때는 null.",       "example": "어떤 글이나 개념에 대해 질문하고 싶은가요?",       "source": "User Flow \- 보완 유도 단계"     },     "improved\_question": {       "type": "string",       "required": false,       "description": "루브릭 기준을 충족한 개선 질문 예시. mode가 completed일 때만 값 있음.",       "example": "'내 마음은 호수요'에서 '호수'가 어떤 점에서 비유 표현인지 설명해 주세요.",       "source": "User Flow \- 목표 달성 단계"     },     "next\_action": {       "type": "string",       "required": true,       "allowed\_values": \[         "ask\_more",         "show\_result"       \],       "description": "프론트엔드가 다음에 수행해야 할 행동. mode와 1:1 대응되며 구현 오류 방지 목적으로 명시.",       "example": "ask\_more",       "source": "User Flow \- 다음 행동 결정 단계"     }   } } |

#### **5.4.2. State Machine**

| 항목 | 내용 |
| :---- | :---- |
| **한 줄 정의** | 서비스의 각 상태와 상태 간 전이 조건, 예외 처리를 정의한 비즈니스 로직 문서 |
| **작성 담당 Agent** | PM Agent |
| **주요 Input** | \[Global\] Constitution / \[Primary\] User Flow, QA Plan / \[Secondary\] Build Plan, MVP Scope |
| **작성 원칙** | ※ PM Agent가 서비스 플로우 관점에서 상태를 정의하고, Tech Agent가 기술적 구현 가능성을 검토하는 구조를 유지한다. PM이 state\_machine을 작성함으로써 Tech Agent의 셀프 검토(Build Plan 작성 후 자기 검토)를 방지한다. **1\. 작성 기준 (핵심)** Constitution ⑦ 루브릭 기반 서비스 플로우의 시스템 판단 구조를 상태값으로 구체화한다. User Flow의 각 화면 전환 조건을 상태 전이 규칙으로 변환한다. QA Plan의 엣지 케이스와 에러 시나리오를 예외 상태로 포함한다. 모든 조건 분기(양호/미흡/예외)를 빠짐없이 명시한다. 추론으로 채울 수 있는 케이스도 생략하지 않는다. **2\. 상태 목록 정의 규칙** 서비스에 등장하는 모든 상태를 나열한다. 각 상태는 현재 사용자 상황을 기준으로 명명한다. 사용자에게 노출되는 상태는 data\_schema의 mode 값과 1:1 대응하여 명시한다. 내부 이동 상태는 mode 값을 부여하지 않고 —로 표기한다. **\[작성 형식\]** | 상태명 | 의미 | 대응 mode 값 | 상태명 | 현재 사용자 상황 설명 | mode 값 또는 — **3\. data\_schema mode 값 매핑 규칙**  상태 목록에서 정의한 각 mode 값이 반환하는 output 필드를 명시한다. 반환 필드는 data\_schema의 output 필드와 일치해야 한다. 사용자에게 노출되지 않는 내부 이동 상태는 반환 없음으로 표기한다.  **\[작성 형식\]**  | mode 값 | 반환 필드 \[mode 값\] | \[반환 필드명 나열\] — (내부 이동) / 반환 없음  **4\. 상태 전이 규칙 정의 규칙** 모든 조건은 IF-THEN 구조로 작성한다. ⑦의 판단 순서(순차/병렬)를 명시적으로 반영하여 양호/미흡 전 케이스를 모두 포함한다. 각 상태의 출력 필드는 data\_schema의 output 필드와 일치해야 한다. 출력이 없는 내부 이동 상태는 출력 블록을 생략한다. **\[작성 형식\]** IF \[조건\] → THEN 상태: \[상태명\] → 출력: 　　mode: \[mode 값\] 　　\[필드명\]: \[값 또는 설명\] 　　next\_action: \[next\_action 값\] **5\. 예외 처리 정의 규칙** QA Plan의 엣지 케이스와 에러 시나리오를 빠짐없이 포함한다. 각 예외 상태에서 사용자에게 보여지는 메시지를 함께 정의한다. 최대 재시도 횟수 초과 시의 처리 규칙을 반드시 명시한다.            **\[작성 형식\]** IF \[예외 조건\] → THEN 상태: \[상태명\] → 출력: 　　\[필드명\]: \[값 또는 메시지\] **6\. 포함하지 않는 내용** 실제 DB 상태 관리 로직 및 서버 내부 처리 방식은 포함하지 않는다. 구현 방법(코드 레벨)은 포함하지 않는다. 구현 방식은 Build Plan을 참조한다. |
